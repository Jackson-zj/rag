# 注册登录与角色权限持久化改造提案

## 目标

在当前 Enterprise RAG MVP 中增加基于 PostgreSQL 的注册、登录、角色分配和知识库访问权限能力，使系统从“内存 demo 账号”升级为可重启后保留用户、角色、权限、文档元数据和聊天记录的本地企业知识库应用。

本次改造还需要调整前端体验：

- 管理员登录后进入管理工作台，可以上传文档、查看知识库、管理用户角色和权限，并使用问答能力。
- 普通用户登录后进入简化问答界面，只看到聊天框和自己的问答记录，不能上传文档或进入管理功能。
- 不同角色只能检索和问答其被授权访问的知识库内容。

## 非目标

- 不实现生产级单点登录、OAuth、短信或邮箱验证。
- 不实现多租户隔离、组织架构、部门树或复杂审批流。
- 不实现细粒度到“单个段落”的人工授权；本阶段以知识库级权限为主。
- 不实现完整审计系统，只保留必要的创建、更新时间字段。
- 不重构 AI 服务的核心检索算法，除非权限数据契约需要小范围适配。

## 用户与运行环境

目标用户包括：

- 系统管理员：负责上传文档、维护用户、创建或分配角色、配置角色可访问的知识库。
- 普通用户：登录后直接向知识库系统提问，只能获取其角色允许访问的内容。

运行环境保持当前三服务架构：

- `frontend`：React + TypeScript + Vite。
- `backend-java`：Spring Boot API，负责认证、授权、用户角色、知识库元数据、文档元数据、聊天会话和 SSE 聚合。
- `ai-service`：FastAPI RAG 服务，负责文档切分、向量索引、权限过滤检索和模型问答。
- PostgreSQL/pgvector：持久化用户、角色、知识库、文档、向量块、会话和消息。

## 功能需求

### 认证与注册

- 系统提供注册接口，普通用户可以自行注册。
- 注册用户默认获得普通用户角色。
- 系统提供登录接口，登录成功后返回 bearer token 和用户信息。
- 密码使用 BCrypt 哈希后写入 PostgreSQL。
- 保留初始化账号：
  - `admin / admin123`：管理员。
  - `analyst / analyst123`：普通或分析类用户，用于演示受限知识库访问。

### 用户、角色与权限

- 管理员可以查看用户列表。
- 管理员可以禁用用户。
- 管理员可以重置用户密码。
- 管理员可以创建角色、查看角色列表，并给用户分配或移除角色。
- 角色包含名称和描述字段。
- 管理员可以给角色分配可访问的知识库。
- 一个用户可以拥有多个角色。
- 用户可访问知识库集合等于其所有角色权限的并集。
- 管理员默认拥有全部知识库访问权限，并拥有上传、管理用户和管理权限的能力。

### 知识库与文档

- 知识库元数据写入 PostgreSQL。
- 文档元数据写入 PostgreSQL。
- 只有管理员可以上传文档。
- 上传文档时，后端根据知识库权限生成 AI 索引所需的访问控制信息。
- 文档去重仍保留当前基于内容哈希的行为。

### 问答与会话

- 普通用户登录后自动进入聊天界面。
- 普通用户不显示上传、知识库编辑、角色管理等管理入口。
- 普通用户不显示文档列表。
- 普通用户不能手动选择知识库范围，后端自动使用该用户可访问的全部知识库。
- 聊天会话和消息写入 PostgreSQL，服务重启后可恢复。
- 创建聊天会话时，后端只允许使用当前用户可访问的知识库。
- 如果前端不提供知识库范围，后端默认使用当前用户可访问的全部知识库。
- 问答检索必须只命中当前用户有权限访问的知识库内容。

### 前端界面

- 登录页支持登录与注册。
- 管理员视图保留并整理当前已有上传文档、知识库列表、问答和事件时间线能力。
- 管理员视图新增用户与角色管理区域。
- 普通用户视图只保留聊天主流程，可查看问答输出和必要状态提示。
- 前端根据登录用户角色控制可见功能，同时以后端权限校验为准。

## 输入与输出

### 输入

- 用户注册信息：用户名、密码。
- 用户登录信息：用户名、密码。
- 管理员操作：角色名称、用户角色分配、角色知识库权限分配。
- 文档上传信息：知识库 ID、文件名、文档文本内容。
- 问答信息：用户问题、可选会话 ID、可选知识库范围。

### 输出

- 登录响应：token、用户 ID、用户名、角色、可访问知识库 ID。
- 用户与角色管理响应：用户列表、角色列表、角色权限配置。
- 文档响应：文档 ID、知识库 ID、文件名、索引状态、创建时间。
- 聊天响应：会话、历史消息、SSE token/tool/done 事件。

## 依赖与约束

- Java 后端需要新增 PostgreSQL 访问依赖和 BCrypt 密码依赖。
- 当前后端代码集中在单个 Java 文件中，允许在详细设计阶段决定是否拆分为多个类以控制复杂度。
- PostgreSQL 初始化 SQL 已存在，需要在兼容现有表结构的基础上增补角色关联、权限、会话和消息表。
- AI 服务已经支持 PostgreSQL 向量存储和 `allowed_user_ids` 过滤；如果后端按角色解析出允许用户 ID，可以尽量少改 AI 服务。
- 当前项目以本地 MVP 为主，不引入外部认证服务。
- 不把明文密码、API key 或本地私密配置写入仓库。

## 成功标准

- 新注册用户写入 PostgreSQL，并可以登录。
- 用户密码不以明文存储。
- 管理员可以禁用用户，被禁用用户不能登录或继续访问接口。
- 管理员可以重置用户密码。
- 服务重启后，用户、角色、权限、知识库、文档元数据、会话和消息仍然存在。
- 管理员可以上传文档，普通用户不能上传文档。
- 管理员可以给用户分配角色，并给角色配置知识库权限。
- 普通用户只能检索和回答其角色允许访问的知识库内容。
- 前端能根据管理员和普通用户展示不同界面。
- 后端测试覆盖登录、注册、权限校验、管理员上传限制和聊天知识库范围限制。
- 前端构建通过。

## 验收检查

- 使用 `admin / admin123` 登录，可以看到管理工作台和上传入口。
- 使用普通用户登录，只看到聊天界面，看不到上传入口。
- 普通用户直接调用上传接口会被后端拒绝。
- 管理员给普通用户分配某个知识库权限后，该用户可以基于该知识库问答。
- 移除该权限后，该用户不能再检索到该知识库内容。
- 管理员禁用用户后，该用户不能登录；已有 token 访问接口也会被拒绝。
- 管理员重置用户密码后，用户可以使用新密码登录。
- 重启 Java 后端后，用户、角色、权限和聊天记录仍可查询。
- 执行 Java 测试通过。
- 执行前端构建通过。

## 假设

- 开放注册仅创建普通用户，不允许注册为管理员。
- 管理员账号由初始化 SQL 或应用启动种子数据创建。
- 权限粒度按知识库控制，不按单个文档控制。
- 管理员默认拥有全部知识库权限。
- 普通用户默认使用其可访问的全部知识库问答。
- 本阶段使用 bearer token，token 可以继续存储在 Java 后端内存中；用户、角色和权限必须持久化。是否持久化 token 留到详细设计阶段评估。
- 聊天会话和消息纳入 PostgreSQL 持久化范围。

## 待确认问题

无。以下决策已确认：

- 允许管理员禁用用户和重置密码。
- 角色需要描述字段。
- 普通用户不需要自己选择知识库，由系统自动使用其全部可访问知识库。
- 普通用户不需要查看文档列表，只通过问答使用知识库。
