# 注册登录与角色权限持久化高层设计

## 架构概览

本次改造保留现有三服务架构，由 Spring Boot 后端成为认证、授权和业务持久化的中心：

- `frontend` 负责登录、注册、管理员工作台和普通用户聊天界面。
- `backend-java` 负责用户注册登录、密码哈希、token 校验、角色管理、知识库权限、文档上传权限、聊天会话持久化和 SSE 转发。
- `ai-service` 继续负责文档切分、向量索引、权限过滤检索和模型问答。
- PostgreSQL/pgvector 持久化用户、角色、权限、知识库、文档元数据、向量块、聊天会话和消息。

核心原则：

- 前端只做体验层权限控制，所有安全判断必须由后端执行。
- 角色权限以“知识库”为粒度，用户可访问知识库为其所有角色权限的并集。
- 管理员拥有全部知识库权限，并拥有上传文档、管理用户、管理角色和配置权限的能力。
- 普通用户不选择知识库，后端自动将其可访问的全部知识库用于问答。

## 模块边界

### 前端

前端拆成三个主要视图状态：

- 未登录视图：登录和注册表单。
- 管理员视图：上传文档、知识库列表、用户管理、角色管理、问答区域和事件时间线。
- 普通用户视图：简化聊天界面和问答状态提示。

前端保存当前 token 和用户信息，并在请求中携带 `Authorization: Bearer <token>`。前端根据用户角色决定显示哪个视图，但不承担最终授权责任。

### Java 后端

Java 后端从当前 `DemoStore` 内存模型迁移到 PostgreSQL 持久化模型。建议在详细设计阶段拆分以下责任：

- 认证服务：注册、登录、密码哈希校验、token 生成和当前用户解析。
- 用户服务：用户列表、禁用用户、重置密码、用户角色分配。
- 角色服务：角色创建、角色列表、角色知识库权限配置。
- 知识库服务：知识库查询和创建。
- 文档服务：管理员上传校验、文档元数据写入、AI 索引调用、RabbitMQ 可选异步通知。
- 聊天服务：会话创建、消息读取、最近 N 轮历史恢复、SSE 问答流转发和消息持久化。
- 权限服务：判断管理员能力、计算当前用户可访问知识库集合。

### AI 服务

AI 服务优先保持现有能力：

- `/ai/documents/index` 接收文档内容、知识库 ID 和访问控制信息。
- `/ai/rag/search` 按知识库 ID 和用户 ID 过滤可见 chunk。
- `/ai/chat/stream` 根据后端传入的用户 ID 和知识库 ID 列表执行问答。

如果 Java 后端继续在上传时向 AI 服务写入 `allowed_user_ids`，AI 服务可以少改。但角色权限变化后，已索引 chunk 的 `allowed_user_ids` 可能过期。因此高层设计建议调整为：

- chunk 只持有 `knowledge_base_id`。
- 查询时以后端传入的 `knowledge_base_ids` 作为主要权限边界。
- `allowed_user_ids` 保留兼容，但不再作为角色权限的唯一来源。

这样角色权限变更后不需要重新索引文档。

## 数据流

### 注册与登录

1. 用户在前端提交用户名和密码。
2. 后端注册时校验用户名唯一性，使用 BCrypt 写入密码哈希，并分配默认普通角色。
3. 登录时后端读取用户，检查是否被禁用，校验 BCrypt 密码。
4. 登录成功后生成 bearer token，返回用户信息、角色和可访问知识库 ID。
5. 后续请求通过 token 找到当前用户，并实时检查用户是否被禁用。

### 管理员配置权限

1. 管理员登录后获取用户列表、角色列表和知识库列表。
2. 管理员创建角色或更新角色描述。
3. 管理员为用户分配角色。
4. 管理员为角色分配知识库权限。
5. 后端根据数据库中的用户角色和角色权限计算用户可访问知识库集合。

### 文档上传与索引

1. 管理员选择知识库并上传文档。
2. 后端校验当前用户是管理员。
3. 后端写入或更新文档元数据，计算内容哈希。
4. 后端调用 AI 服务同步索引文档。
5. AI 服务将 chunk 和 embedding 写入 PostgreSQL/pgvector。
6. 后端记录文档状态；RabbitMQ 保留为可选异步路径，但同步索引结果仍作为本地 MVP 的权威结果。

### 普通用户问答

1. 普通用户登录后进入聊天界面。
2. 前端创建会话时可以不传知识库 ID。
3. 后端计算用户可访问的全部知识库，并写入会话范围。
4. 用户提问后，后端保存用户消息。
5. 后端向 AI 服务发送 `user_id`、`session_id`、问题和可访问知识库 ID 列表。
6. AI 服务只在这些知识库内检索，并通过 SSE 返回工具调用和 token。
7. 后端转发 SSE，并在完成后保存助手消息。

## 存储设计

现有 PostgreSQL 表会扩展为以下逻辑模型：

- `users`：用户账号、密码哈希、禁用状态、创建时间、更新时间。
- `roles`：角色名称、描述、是否系统内置。
- `user_roles`：用户和角色的多对多关系。
- `role_knowledge_bases`：角色和知识库的多对多权限关系。
- `knowledge_bases`：知识库元数据。
- `documents`：文档元数据、状态、内容哈希、chunk 数量。
- `document_chunks`：向量块、内容、embedding 和知识库 ID。
- `chat_sessions`：会话归属用户、标题和创建时间。
- `chat_session_knowledge_bases`：会话使用的知识库范围。
- `chat_messages`：会话消息、角色、内容和创建时间。

token 可以先继续存储在 Java 内存中，但鉴于“禁用用户后已有 token 也要被拒绝”，后端每次 token 解析后必须从数据库读取用户状态。是否将 token 也落库留到详细设计阶段决定。

## 外部依赖

Java 后端需要新增：

- PostgreSQL JDBC 驱动。
- Spring JDBC 或 JdbcTemplate 相关依赖。
- Spring Security Crypto，用于 BCrypt。

不建议在本次 MVP 中引入完整 Spring Security 过滤链，原因是当前项目控制器结构较轻，使用显式 token 校验更贴近现有代码，变更面更小。详细设计中仍需保证所有受保护接口都统一调用认证和权限校验。

## 主要设计决策

- 采用知识库级 RBAC，而不是文档级或 chunk 级 ACL。
- 普通用户默认问答其全部可访问知识库，避免普通用户界面复杂化。
- 管理员上传功能只由后端角色校验控制，前端隐藏入口只是辅助体验。
- 密码使用 BCrypt，避免明文密码进入数据库。
- 角色权限变更不触发文档重新索引，查询时由知识库范围控制权限。
- 聊天会话和消息持久化到 PostgreSQL，避免后端重启丢失记录。
- 用户登录后默认恢复其最近会话的最新 10 轮问答，且历史读取仍受会话归属校验限制。

## 备选方案

### 使用 Spring Security 全套认证授权

优点是安全能力更完整，过滤链更标准。缺点是当前项目较小，改造范围会明显扩大，也会压缩业务功能实现时间。本阶段建议暂不采用。

### 将权限继续写入 `allowed_user_ids`

优点是兼容 AI 服务当前实现。缺点是角色权限变更后旧 chunk 权限会过期，需要重建索引或批量更新 chunk。建议保留字段兼容，但查询主边界改为知识库 ID。

### 不持久化聊天记录

优点是实现更快。缺点是不满足本次“写入 PostgreSQL”和重启后保留记录的目标。当前提案不采用。

## 风险与未知

- 当前 Java 后端集中在单个文件中，继续堆叠功能会降低可维护性；详细设计应明确拆分策略。
- 如果本地 PostgreSQL 不可用，后端无法再完整运行注册登录功能；需要保留清晰的连接配置和错误提示。
- 现有 README 和部分源码中的中文显示可能存在编码问题，后续改动要避免继续扩大乱码。
- AI 服务中 `allowed_user_ids` 的兼容逻辑需要谨慎处理，避免权限判断同时存在两套互相冲突的来源。
- 管理员禁用用户后，内存 token 如果不清理，也必须在每次请求时重新检查数据库用户状态。

## 高层测试策略

- 后端单元或集成测试覆盖注册、登录、禁用用户、重置密码、角色分配、知识库权限计算和上传权限拒绝。
- 后端测试覆盖普通用户创建会话时自动使用可访问知识库。
- 后端测试覆盖被移除权限后无法再创建包含该知识库的会话。
- AI 服务测试覆盖按知识库 ID 检索过滤，不因 `allowed_user_ids` 为空而越权。
- 前端执行 TypeScript 和 Vite 构建，验证管理员视图和普通用户视图类型正确。
- 端到端烟测使用管理员上传文档、普通用户问答、权限移除后问答不可见三个路径。
